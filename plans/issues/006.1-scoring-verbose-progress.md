# Issue 006.1 — Verbose Scoring Progress

## Goal

Make LLM scoring observable during long transcript scoring runs without compromising automation-friendly CLI output. When `clipper score` is run with `--verbose`, show lifecycle diagnostics, model/config details, window-based progress, retry/warning counts, and token usage when the OpenAI-compatible API provides it. All verbose output must go to stderr so stdout remains reserved for human command results or parseable JSON envelopes.

## Depends On

- Issue 006

## Tasks

- Extend the shared progress/logging abstraction introduced for transcription so scoring can write verbose diagnostics and progress to stderr without depending directly on argparse.
- Thread the existing CLI verbosity setting into the score command/library boundary.
- Emit verbose lifecycle messages for scoring, including:
  - resolved video name and transcript path
  - scoring directive summary
  - LLM base URL host/origin, model name, temperature, and timeout
  - transcript duration and segment count
  - number of transcript scoring windows/chunks to process
  - score artifact output path
- Add window-based progress behind `--verbose`:
  - report current window number, total windows, and percentage completion
  - include each window's timestamp range when useful
  - keep progress deterministic and testable from the number of transcript windows, not from token streaming
- Keep `--json` stdout strictly parseable JSON. All verbose logs, progress indicators, retry notes, warnings, and token summaries must go to stderr.
- Preserve non-verbose behavior: no lifecycle chatter or progress output should appear unless `--verbose` is passed.
- Capture token usage from OpenAI-compatible responses when available. If the response includes usage metadata, report at least:
  - prompt tokens
  - completion tokens
  - total tokens
  - cumulative totals across all windows and retries
- Do not estimate tokens locally when usage metadata is absent. Instead, verbose output should clearly indicate that token usage was not provided by the API.
- Preserve the existing retry behavior for invalid JSON. In verbose mode, log retry events with the affected window range and include retry token usage in cumulative totals when returned by the API.
- Preserve existing validation/warning behavior. In verbose mode, log a concise warning summary, including dropped/clamped segment counts if this can be derived cleanly from existing warning messages.
- Preserve existing `--reuse` behavior: reused scores should not instantiate/call the LLM and should not show scoring progress, but may log that existing scores were reused when verbose.
- Preserve existing failure behavior and JSON/human error envelopes. If an LLM request, parsing, or validation step fails, verbose output may show the last completed progress step, but command exit/error behavior must remain consistent.
- Ensure progress display degrades cleanly when stderr is not an interactive TTY:
  - no animated/control-character progress bar in non-TTY contexts
  - plain verbose milestone lines are acceptable
- Avoid printing full prompts, transcript text, or raw model responses by default, even under `--verbose`, to avoid leaking transcript/private content into logs.
- Add tests proving:
  - non-verbose scoring remains quiet apart from the final command result
  - verbose scoring writes lifecycle/config/progress information to stderr
  - `--json --verbose` writes valid JSON to stdout and logs only to stderr
  - window progress is based on processed transcript windows and reaches 100% for mocked LLM responses
  - model name, temperature, timeout, transcript segment count, and window count are visible in verbose stderr
  - token usage is accumulated and logged when mocked responses include OpenAI-style usage metadata
  - verbose output clearly reports unavailable token usage when mocked responses omit usage metadata
  - retry token usage is included in totals when an invalid JSON response triggers a retry and the retry response includes usage
  - reused scores do not call the LLM and do not show scoring progress
  - non-TTY stderr does not emit animated progress control sequences
- Document verbose scoring progress in `README.md`, including:
  - use `uv run clipper score VIDEO --directive "..." --verbose`
  - progress and diagnostics are written to stderr
  - stdout remains parseable under `--json`
  - token usage appears only when returned by the configured OpenAI-compatible endpoint

## Acceptance Criteria

- `uv run clipper score VIDEO --directive "Find compelling clips" --verbose` shows useful stderr lifecycle output, selected model/config, window-based progress with percentage completion, and final output path.
- `uv run clipper score VIDEO --json --verbose` keeps stdout parseable as a single JSON envelope while writing verbose logs/progress only to stderr.
- Non-verbose scoring output remains unchanged and does not show progress or lifecycle chatter.
- Verbose scoring reports token usage totals when the mocked/OpenAI-compatible response includes usage metadata, and clearly reports when usage metadata is unavailable.
- Progress reporting is tested with mocked LLM responses and does not require real LLM calls in the default test suite.
- Reuse mode remains fast and does not instantiate/call the LLM when valid scores are reused.
- README explains how to use `--verbose` to observe long-running scoring and clarifies that token usage depends on API-provided usage metadata.
